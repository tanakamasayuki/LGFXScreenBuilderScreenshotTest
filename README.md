# LGFXScreenBuilder Screenshot Test

Screenshot fixture repo for [LGFXScreenBuilder](https://github.com/tanakamasayuki/LGFXScreenBuilder):
it renders a screen definition (`Sfm.lgfxsb.json` → `Sfm.h`) on the headless
`lang-ship:host` SDL backend and publishes a gallery of **every profile ×
every scene**.

It is intentionally capture-only — there is no application sketch here. The two
files that matter are the project definition and its generated header; runnable
device examples live in the main repo's `examples/`.

The gallery is regenerated by CI and served from `docs/` via GitHub Pages:

- **By profile** (`docs/index.html`) — every scene for each device profile.
- **By scene** (`docs/by-scene.html`) — every profile side-by-side per scene,
  for spotting layout differences across devices.

## Layout

```
Sfm.lgfxsb.json            # project definition (profiles + scenes) — source of truth
Sfm.h                      # generated header — the capture's only real input
screenshot/                # headless capture harness
  screenshot.ino           #   renders all profiles × scenes to output/*.png
  sketch.yaml              #   lang-ship:host:host:mode=lgfx profile
  test_screenshot.py       #   pytest-embedded driver
gen_gallery.py             # builds docs/ gallery from the captured PNGs (stdlib only)
docs/                      # committed gallery (GitHub Pages source)
  shots/<profile>/<scene>.png
```

## How it works

1. `screenshot.ino` includes the generated `../Sfm.h`, creates an off-screen
   `LGFX` device sized to each profile's native resolution, calls
   `screen.show(sceneId)` (design/preview state), and writes the framebuffer
   with `createPng()` to `output/<profile>__<scene>.png`. One run captures the
   whole matrix regardless of the physical board.
2. `test_screenshot.py` builds and drives the sketch through
   [pytest-embedded-arduino-cli](https://github.com/tanakamasayuki/pytest-embedded-arduino-cli)
   on the headless `mode=lgfx` backend (SDL dummy video driver).
3. `gen_gallery.py` copies the PNGs under `docs/shots/` and emits the two
   static HTML pages — Python standard library only, no Node, no dependencies.

## Run locally

Requires `arduino-cli`, `uv`, and SDL2 (`sudo apt-get install -y libsdl2-dev`).
The [`LGFXScreenBuilder`](https://github.com/tanakamasayuki/LGFXScreenBuilder)
library repo must sit next to this one (sibling directory), matching
`sketch.yaml`'s `dir: ../../LGFXScreenBuilder`.

```bash
uv run pytest screenshot/ -v   # capture → screenshot/output/*.png
python3 gen_gallery.py         # gallery → docs/
```

Open `docs/index.html` in a browser. **GitHub Actions is optional** — the Action
just automates these two commands.

## Use it in your own project

The harness only depends on the generated header plus the LGFXScreenBuilder
library, so it drops into any project. Setup is a one-time step; after that you
only ever touch the header.

1. Copy the file set into a new repo: `*.lgfxsb.json`, the generated header,
   `screenshot/`, `gen_gallery.py`, `conftest.py`, `pyproject.toml`,
   `.github/`, `.gitignore`.
2. Delete `docs/` — it is fully regenerated.
3. Point the harness at your header: rename the header to yours and update the
   `#include "../<YourHeader>.h"` line in `screenshot/screenshot.ino` to match
   (the include name must match your generated `.h`).
4. Push. The Action captures every profile × scene, builds `docs/`, and commits
   it back.
5. Settings → Pages → **Deploy from a branch** → `main` / `/docs` → published.
6. From then on, regenerate **only the header** from LGFXScreenBuilder and push
   it — the gallery updates automatically.

### Where to keep the images

Committed PNGs grow `.git` over time. **Prefer a dedicated screenshot repo**
(what this is) so the images never enter your application repo's history.

If you must keep capture inside an existing application repo, isolate the
gallery instead of committing `docs/shots/` to the branch: publish `docs/` to
GitHub Pages via the Pages deploy **artifact** (`actions/deploy-pages`) or push
it to an external store. Periodically squashing history also keeps things lean.

> This sample is intentionally minimal. Feel free to squash its history to keep
> the repo small.

## CI

`.github/workflows/screenshots.yml` runs on pushes that touch the project or
harness (and on demand). It checks out this repo and the library repo as
siblings, captures, regenerates `docs/`, and commits the gallery back.
