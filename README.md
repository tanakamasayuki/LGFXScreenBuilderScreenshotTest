# LGFXScreenBuilder Screenshot Test

🌐 English | [日本語](README.ja.md)

Screenshot fixture repo for [LGFXScreenBuilder](https://github.com/tanakamasayuki/LGFXScreenBuilder):
it renders the generated header `Sfm.h` on the headless `lang-ship:host` SDL
backend and publishes a gallery of **every profile × every scene**.

> 📷 **Live gallery: <https://tanakamasayuki.github.io/LGFXScreenBuilderScreenshotTest/>**

**`Sfm.h` is the single source.** Exported from LGFXScreenBuilder with the
*"Embed AI layouts"* option on, it carries each scene's AI-layout JSON in a
comment block (SPEC §10.2) — stripped at compile, so it costs nothing on the
device but lets this repo derive the whole gallery from the header alone (no
project file). It is intentionally capture-only — there is no application
sketch; runnable device examples live in the main repo's `examples/`.

The project file `Sfm.lgfxsb.json` is **not required** by the gallery. It is
kept in this sample only so the sample `Sfm.h` can be regenerated (re-export
from the tool, or run codegen, with *Embed AI layouts* on). A real adopter does
not need to commit it — see [Use it in your own project](#use-it-in-your-own-project).

The gallery is regenerated by CI and served from `docs/` via GitHub Pages at
<https://tanakamasayuki.github.io/LGFXScreenBuilderScreenshotTest/>:

- **By profile** ([index.html](https://tanakamasayuki.github.io/LGFXScreenBuilderScreenshotTest/index.html)) — every scene for each device profile.
- **By scene** ([by-scene.html](https://tanakamasayuki.github.io/LGFXScreenBuilderScreenshotTest/by-scene.html)) — every profile side-by-side per scene, for spotting layout differences across devices.
- Each scene has a **⧉ AI JSON** button that copies that scene's layout JSON, so
  a wrong-looking screen can be pasted straight into an AI to fix.

## Workflow

This gallery is for **checking the overall design at a glance**. The loop:

1. **Edit** the screens in [LGFXScreenBuilder](https://github.com/tanakamasayuki/LGFXScreenBuilder)
   (the browser tool). In Chromium-based browsers (Chrome / Edge) it writes to
   local files directly via the File System Access API — **Save project** and
   **Export header** overwrite this repo's `Sfm.lgfxsb.json` / `Sfm.h` in place,
   with no download-and-copy step.
2. **Generate** the screenshots, either way:
   - locally — `uv run pytest screenshot/` rebuilds `docs/`, or
   - on push — GitHub Actions regenerates the gallery and commits it back; you
     then just `git pull` to get only the changed files, which is the handy part.
3. **Review** the rough design in the gallery (the by-profile / by-scene pages).
4. **Verify the details elsewhere.** These renders are the static design/preview
   state, so text swaps and anything drawn dynamically at runtime won't appear
   here — confirm those on real hardware, or on a PC via the SDL2 `host` display.

## Layout

```
Sfm.h                      # generated header (with embedded AI layouts) — the single source
Sfm.lgfxsb.json            # project file — NOT needed by the gallery; kept only to regenerate the sample Sfm.h
screenshot/                # headless capture harness
  screenshot.ino           #   renders all profiles × scenes to output/*.png
  sketch.yaml              #   lang-ship:host:host:mode=lgfx profile
  test_screenshot.py       #   pytest-embedded driver + gallery build/verify
  conftest.py              #   wipes output/ before the capture test
  gen_gallery.py           #   builds docs/ gallery by parsing Sfm.h (stdlib only)
docs/                      # committed gallery (GitHub Pages source)
  index.html / by-scene.html / ai-layouts.js
  shots/<profile>/<scene>.png
```

## How it works

1. `screenshot.ino` includes the generated `../Sfm.h`, creates an off-screen
   `LGFX` device sized to each profile's native resolution, calls
   `screen.show(sceneId)` (design/preview state), and writes the framebuffer
   with `createPng()` to `output/<profile>__<scene>.png`. One run captures the
   whole matrix regardless of the physical board.
2. `test_screenshot.py` builds and drives the sketch through
   [pytest-embedded-arduino-cli](https://github.com/tanakamasayuki/pytest-embedded-arduino-cli)
   on the headless `mode=lgfx` backend (SDL dummy video driver).
3. `gen_gallery.py` parses the embedded AI-layout block in `Sfm.h` for scene
   names, descriptions, and profiles, copies the PNGs under `docs/shots/`, and
   emits the two HTML pages plus `ai-layouts.js` (the copy-button payloads) —
   Python standard library only, no Node, no dependencies. `test_screenshot.py`
   imports it and runs this right after capture, so the gallery is built **and
   verified** (every profile × scene present, pages non-empty) in the same run.

**Why LovyanGFX?** Capture renders through
[LovyanGFX](https://github.com/lovyan03/LovyanGFX). Even if your application
draws via **M5Unified / M5GFX**, the drawing API is compatible, so screens come
out identical here. LovyanGFX also lets the off-screen device be sized to any
resolution, which is exactly what makes capturing every profile on a single
headless host convenient.

## Run locally

Requires `arduino-cli`, `uv`, and SDL2 (`sudo apt-get install -y libsdl2-dev`).
No manual library download is needed: `sketch.yaml` pins
[`LGFXScreenBuilder`](https://github.com/tanakamasayuki/LGFXScreenBuilder) (and
LovyanGFX) by version, so arduino-cli auto-installs them on first compile.

```bash
uv run pytest screenshot/ -v   # capture → screenshot/output/*.png, then build + verify docs/
```

That single command captures the shots and regenerates the gallery. Open
`docs/index.html` in a browser. **GitHub Actions is optional** — the Action just
runs this command and commits `docs/` back. (To rebuild only the gallery from
existing shots: `uv run python screenshot/gen_gallery.py`.)

## Use it in your own project

The harness depends only on the generated header (the LGFXScreenBuilder library
needed to build is pinned by version in `sketch.yaml` and auto-installed), so it
drops into any project. Setup is a one-time step; after that you only ever touch
the header.

1. In LGFXScreenBuilder, export the header with **"Embed AI layouts" on** (so
   the gallery can read scenes/profiles/layouts from the `.h` alone).
2. Copy the file set into a new repo: the generated header, `screenshot/` (which
   now contains `gen_gallery.py` and `conftest.py`), `pyproject.toml`,
   `.github/`, `.gitignore`. No project file is needed.
3. Delete `docs/` — it is fully regenerated.
4. Point the harness at your header: rename the header to yours and update the
   `#include "../<YourHeader>.h"` line in `screenshot/screenshot.ino` to match.
5. Push. The Action captures every profile × scene, builds `docs/`, and commits
   it back. Then set Settings → Pages → **Deploy from a branch** → `main` /
   `/docs` → published.
6. From then on, re-export **only the header** (Embed AI layouts on) and push it
   — the gallery updates automatically.

### Where to keep the images

Committed PNGs grow `.git` over time. **Prefer a dedicated screenshot repo**
(what this is) so the images never enter your application repo's history.

If you must keep capture inside an existing application repo, isolate the
gallery instead of committing `docs/shots/` to the branch: publish `docs/` to
GitHub Pages via the Pages deploy **artifact** (`actions/deploy-pages`) or push
it to an external store. Periodically squashing history also keeps things lean.

> This sample is intentionally minimal. Feel free to squash its history to keep
> the repo small.

## CI

`.github/workflows/screenshots.yml` runs on pushes that touch the project or
harness (and on demand). It checks out this repo, captures (the platform and
libraries auto-install from `sketch.yaml` on first compile), regenerates
`docs/`, and commits the gallery back.
